§ 01

Architecture Overview

RhythmFlex AI follows a three-tier client-server architecture running entirely on a local Docker Compose stack. The React SPA provides the employee and manager interface. FastAPI handles all business logic, workflow automation, and AI orchestration. PostgreSQL stores all relational data with full audit trails.

ℹ️
Golden Rule: All AI processing is server-side only. The Anthropic API key never reaches the browser. The React frontend communicates exclusively with the FastAPI backend over JWT-secured REST APIs on localhost.

D2 · Three-Tier Application Architecture

RhythmFlex AI uses a classic three-tier architecture: Presentation, Application, and Data layers. Each tier is a separate Docker service and communicates only with the tier directly below it.

Three-Tier Architecture — Internal ViewLevel 1 · Architecture
Tier 1 · Presentation (React)
Tier 2 · Application (FastAPI)
Tier 3 · Data (PostgreSQL + Redis)
Each tier only communicates downward. React → FastAPI via REST/JSON over HTTP. FastAPI → PostgreSQL via SQLAlchemy ORM. FastAPI → Redis via redis-py. No direct DB access from React.

§ 09

D7 · Security & Authentication Flow

JWT-based authentication with role-based access control enforced at the FastAPI layer. The Anthropic API key is environment-variable-only — it exists only inside the backend container and never reaches the browser or logs.

Authentication & Security FlowSecurity View
JWT access tokens expire in 8 hours. Refresh tokens in 7 days. All tokens stored in React in-memory state (never localStorage). Caregiving requests are hidden from line managers at the RBAC layer — HR Admin role only.
§ 10

Architectural Design Decisions

Key decisions made during architecture design with rationale — documented for traceability and future reference.

ADR-001
FastAPI over Django/Flask
FastAPI provides automatic OpenAPI docs, async support for streaming Claude API responses, and Pydantic v2 for type-safe data models. Critical for the AI streaming feature.
ADR-002
PostgreSQL over MongoDB
The approval workflow is a deeply relational domain (users → requests → approvals → policies). RDBMS with FK constraints and ACID guarantees is the correct choice. The brief explicitly prohibits JSON data input.
ADR-003
Server-side Claude API calls only
API key security is non-negotiable. All AI prompts are constructed server-side with policy context injected from the database. The browser only receives the generated text.
ADR-004
Separate Streamlit service for Code Quality Agent
The brief mandates Streamlit/Gradio for the Code Quality Checker but forbids it for the main UI. Isolating it as a separate Docker service satisfies both requirements cleanly.
ADR-005
Redis for sessions, not JWT blacklist
Redis is used to cache AI responses (reduces repeat API costs) and for session data. Short JWT expiry (8h) avoids the need for a token blacklist for MVP scope.
ADR-006
Docker Compose over Kubernetes
The brief specifies local deployment. Docker Compose provides the simplest reproducible environment with a single command. Kubernetes would be the cloud migration path.
